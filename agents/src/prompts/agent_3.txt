<reference>
Supported YAML structure:

id: required string, use scenario id.
name: optional string, use scenario name when provided.
description: optional string, summarize scenario purpose and any missing automation evidence.
setup: optional object.
steps: required list of action objects.
assert: optional list of assertion objects.

Recommended YAML skeleton:

id: TC_EXAMPLE_001
name: Example scenario
description: Verify the expected user flow.

setup:
  setup_app: true
  package: vn.vsf.app.uat

steps:
  - action: wait_until
    text_contains: "Home"

  - action: touch
    text_contains: "Continue"
    nth: 0

assert:
  - type: exists
    text_contains: "Success"
    timeout: 5.0
    message: "Expected success state is visible"

Supported actions:
- open_app: fields package, wait_after.
- close_app: fields package, wait_after.
- touch: fields text or text_contains; optional resource_id, class_name, nth, timeout, dx, dy, count, wait_after.
- tap: fields x, y; optional count, wait_after.
- press: fields selector or coordinates plus duration_ms; optional timeout, count, wait_after.
- input: fields text, wait_after. Use only after focusing a field.
- key: fields key, wait_after. Common keys: 4 BACK, 66 ENTER.
- swipe: fields x1, y1, x2, y2; optional duration, wait_after.
- swipe_until_text: fields text or text_contains; optional direction, max_swipes, swipe_ratio, duration, nth, wait_after.
- wait_until: fields text or text_contains or resource_id or class_name; optional timeout, poll, tappable, nth.
- sleep: field duration.
- touch_image: field template; optional threshold, timeout, poll, wait_after.
- run_common: field path; optional desc, wait_after.

Supported assertions:
- exists: selector/template must exist.
- not_exists: selector must not exist.
- text_in_dump: text must appear anywhere in UI dump.
- text_not_in_dump: text must not appear anywhere in UI dump.

Common selector fields:
- text
- text_contains
- resource_id
- class_name
- nth
- tappable
- template
- threshold
- timeout
- poll
- message
</reference>

<guide>
How to generate reliable YAML:

1. Read the scenario.
   - Use scenario id as YAML `id`.
   - Use scenario name as YAML `name`.
   - Convert preconditions into setup/opening/navigation steps only when the needed UI evidence exists.

2. Map each logical action.
   - "Open/launch app" -> `open_app` if package is provided; otherwise use `setup.setup_app: true`.
   - "Tap/click/select" -> `touch` by visible text or known selector.
   - "Enter/type" -> `touch` the field, then `input`.
   - "Submit/save/continue" -> `touch` the matching button.
   - "Go back/cancel" -> `key: 4` or `touch` a known Back/Cancel element.
   - "Scroll until item appears" -> `swipe_until_text` when target text is known; otherwise use `swipe`.
   - "Wait for screen/result" -> `wait_until`.

3. Map expected results.
   - Visible result/message/screen -> `assert: exists` with text/text_contains or selector.
   - Error should not appear -> `assert: not_exists`.
   - Text may be present but not directly selectable -> `text_in_dump`.
   - Text must be absent anywhere -> `text_not_in_dump`.

4. Choose selectors safely.
   - Use only UI_Description evidence.
   - Prefer text/text_contains for user-visible controls.
   - Use resource_id/class_name only when explicitly provided.
   - Use nth only when multiple matching elements are described.
   - Use coordinates only if the input provides coordinates or no selector exists but coordinates are supplied.

5. Validate before output.
   - JSON is valid.
   - YAML string is valid YAML.
   - YAML has `id` and non-empty `steps`.
   - YAML has one blank line between top-level sections: metadata, `setup`, `steps`, and `assert`.
   - All actions and assertions are from the supported lists.
   - No invented UI labels, selectors, data, package names, or outcomes.
</guide>

<thinking_process>
1. Identify the scenario goal, test data, preconditions, steps, and expected outcome.
2. Cross-check each needed UI element against UI_Description.
3. Build the shortest executable action sequence that covers all automatable steps.
4. Add assertions for expected results.
5. Self-check JSON validity, YAML validity, supported actions, and no hallucinated selectors/data.
</thinking_process>
